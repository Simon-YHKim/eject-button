# Bilingual Interaction Rules (Korean / English)

Skill-Gen Agent is designed to be usable by Korean speakers while
producing skills that trigger reliably under Claude's English-leaning
matcher.

## Detection

Detect the user's primary language from their **first message**.

- Contains Hangul (`[가-힣]`) → Korean user.
- Otherwise → English user.

Mixed messages (Korean with English tech terms) → Korean user.

## Rules

### For Korean users

- **Chat in Korean.** Questions, confirmations, explanations, progress
  updates — all Korean. Use polite form (`~해요`, `~합니다`).
- **Write artifacts in English.** `SKILL.md`, code, commit messages,
  CHANGELOG, reference files, script docstrings — English.
- **Trigger phrases in descriptions:** include **both** English and
  Korean verbatim phrases. Example:

  ```yaml
  description: >
    Use when the user wants to convert Excel files to CSV.
    Triggers on phrases like "convert xlsx", "export to csv",
    "엑셀 변환", "csv로 바꿔줘". Produces a new .csv file.
  ```

- **Explain the why in Korean** when presenting the file plan, but
  keep file names, code, and frontmatter English.

### For English users

- Everything in English. Do not translate phrases into Korean unless
  the user asks.

## Rationale

Claude's skill matcher responds more reliably to English descriptions,
but end users want to talk in their own language. The solution is to
decouple **the UI** (the chat) from **the artifact** (the skill
package). Chat in the user's language; ship artifacts in English.

Including Korean verbatim phrases in the `description` is important
because Korean users will type Korean trigger phrases, and the matcher
does handle Korean when it's explicitly present in the description.

## Do not

- Do not machine-translate technical terms (e.g. "frontmatter" stays
  "frontmatter").
- Do not write `SKILL.md` body in Korean "just this once" — it will
  under-trigger and the skill will look abandoned.
- Do not commit messages in Korean; keep git history English for
  searchability.
